# Progress Storage Plan

_Last updated: 2026-03-15_

## 1. Current snapshot
- Progress хранится через `ProgressRepository`. Сейчас в дев-среде используется файл (`backend/var/progress.json`), следующая остановка — Supabase Postgres (free tier).
- Одно сохранение содержит: `userId`, `puzzleId`, `grid` (JSON массива 0/1/-1), `mistakes`, `completed`, `updatedAt`.
- Фронт шлёт autosave сразу при первом действии и не чаще 1 раза/с дальше (leading + trailing throttle).

## 2. Growth assumptions
| Параметр | Значение |
| --- | --- |
| Размер одной записи | ~0.3 KB (UUID + ids + JSON + метаданные) |
| Прогрессов на MAU | 10–20 (пользователь решает 10–20 пазлов/мес) |
| Writes в пике | ≈1 write/сек/активный пользователь |

### Емкость по тарифам Supabase
- **Free (500 MB, 15 коннектов):** ~1.6 млн progress-записей ⇒ 30–50k MAU. До 50 write/s.
- **Pro (8 GB, 50 коннектов):** ~28 млн записей ⇒ 250–300k MAU. 200–300 write/s.
- **Team (50 GB+, 100+ коннектов):** сотни миллионов записей, тысячи write/s.

## 3. Retention & archiving
| Тип данных | Правило | Причина |
| --- | --- | --- |
| Прогресс гостя | Удалять, если `updatedAt` > 60 дней | Гость редко возвращается спустя >2 мес; экономим место |
| Прогресс зарегистрированного пользователя | Хранить only last state per puzzle | Этого достаточно, чтобы продолжить; историю можно восстановить из аналитики |
| Завершённые пазлы | Через 30 дней переносить в агрегат (fact table) + удалять сетку | После завершения достаточно отметки “done” + таймштампа |
| Сырые сетки для аналитики | Раз в месяц выгружать в S3 (Parquet) и чистить в OLTP | Снимает нагрузку с основной БД |

## 4. Формат хранения / оптимизация
- **Short term:** JSONB массив `int2[]` (как сейчас) — быстро внедряется, удобно дебажить.
- **Mid term:** Bitset (e.g., `bit(256)`), mistakes `int2`, completed `bool`. Даёт ~5× экономию.
- **Long term:** Партиционирование по `user_id` hash либо по дате `updated_at` (monthly). Позволяет делать VACUUM/backup точечно.

## 5. Ops & мониторинг
- **Метрики:**
  - disk usage (`pg_database_size`),
  - число записей в `puzzle_progress`,
  - avg/write latency,
  - количество открытых коннектов,
  - доля “stale” прогрессов (>60 дней).
- **Фоновые job'ы:**
  1. `cleanup_guest_progress` (daily) — удаляет гостевые записи по TTL.
  2. `archive_completed_progress` (weekly) — переносит завершённые пазлы в агрегат и чистит сетку.
  3. `export_progress_snapshot` (monthly) — выгружает Parquet в S3 для аналитики.

   - Добавить поля `createdAt` / индекс по `updatedAt`.
   - Завести этот doc как живой (review раз в квартал).
2. **После переезда на Pro**
   - Включить bitset-формат сетки.
   - Настроить TTL job для гостевых прогрессов.
3. **Team / высокий трафик**
   - Партиционировать таблицу (по месяцу).
   - Вынести завершённые пазлы в отдельную таблицу `puzzle_completion`.
   - Настроить replica для аналитики.
